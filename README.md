1. 개요
    본 프로젝트는 오늘의집 Android 앱 로그인 기능에 대한 QA 자동화 과제로,
    Python과 Appium을 활용하여 로그인 주요 시나리오를 자동화 테스트 형태로 구현하였다.

    과제 범위 내에서 단순 구현에 그치지 않고, 테스트 시나리오 선정 기준, 자동화 대상 판단 이유, 테스트 구조 설계 의도를 함께 고려하여 구성하였다.

2. 자동화 환경 선택 이유
    [Android 선택 사유]
        개인 개발 환경상 macOS 및 iOS 실기기를 보유하고 있지 않아, 
        Android Emulator 기반의 Appium 자동화 환경이 가장 현실적인 선택지라 판단하였다.

        또한 Android Emulator 환경은 과제 수행 및 반복 실행 시 비교적 안정적인 테스트 환경을 제공할 수 있어,
        Android 앱 기준으로 자동화 테스트를 진행하였다.

3. 테스트 전략 및 설계 방향
    1) 테스트 독립성 확보
        - 각 테스트 케이스는 서로 상태를 공유하지 않도록 독립적으로 실행
        - 테스트 시작 시마다 Appium driver를 새로 생성하여 앱을 실행
        - 특정 케이스 실패 시에도 다음 테스트가 중단되지 않도록 설계
        - 각 테스트는 단독 실행 가능하도록 설계되어 있으며, 실패 여부를 개별적으로 확인 가능

    2) 실패 원인 파악 용이성
        - 각 테스트 케이스는 개별 함수로 구성
        - 실패 시 해당 테스트 케이스 단위에서 즉시 원인 확인 가능
        - 공통 UI 동작은 모듈로 분리하여 가독성과 유지보수성 확보

    3) 확장 가능성을 고려한 현실적인 설계
        - 실무 환경에서도 확장 가능하도록 테스트 구조를 설계하고자 하였으나, 과제의 목적과 범위를 고려하여 과도한 프레임워크 구성은 지양
        - 대신 테스트 구조의 명확성, 유지보수 용이성, 테스트 의도 전달에 집중하여 과제 범위 내에서 현실적인 수준의 자동화 구조를 구성

4. 프로젝트 구조
    AUTOMATE/
    ├─ actions/
    │  ├─ element.py        # UI 요소 관련 공통 액션 (click, input, scroll 등)
    │  └─ app_action.py     # 앱 생명주기 제어 (앱 재실행 등)
    ├─ driver.py            # Appium driver 생성 및 capability 설정
    ├─ settings.py          # 공통 설정값 (패키지명, wait 등)
    ├─ test_login.py        # 로그인 테스트 시나리오
    ├─ run.py               # 테스트 실행 엔트리 포인트
    ├─ .env                 # 테스트 계정 정보 (gitignore 처리)
    └─ .gitignore

    [구조 설계 의도]
        - driver / action / test 역할을 명확히 분리
        - UI 동작 로직과 테스트 시나리오를 분리하여 수정 및 확장 용이
        - 테스트 코드 가독성을 높여 실패 지점 추적이 쉬운 구조로 구성


7. 자동화 구현 시 고려 사항 및 제한
    [동적 UI 요소 처리]
        일부 UI 요소(예: 로그아웃 버튼)는 고유 resource-id / accessibility-id가 제공되지 않아
        XPath 또는 UiScrollable 기반 접근 방식을 사용하였다.

        로그아웃 UI는 동적 구조로 구성되어 있어 식별자 기반 자동화의 안정성 확보에 한계가 있으며,
        실무 환경에서는 테스트 안정성 향상을 위해 고유 식별자 추가가 필요하다고 판단된다.

8. 테스트 실행 방법
    python run.py


    실행 시 모든 로그인 관련 테스트 케이스가 순차적으로 실행되며,
    각 테스트의 PASS / FAIL 여부가 콘솔 로그로 출력된다.
